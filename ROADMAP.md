# Tranzor Translation Exporter — 产品路线图

> 从"翻译导出工具"到"L10n PM 日常工作台"的四阶段演进规划

---

## 现状基线

| 维度 | 当前状态 |
|------|---------|
| 核心功能 | 4 个标签页：File Translation 导出、MR Pipeline 查询导出、Quality Overview 可视化、Full Translation 全量导出 |
| 数据架构 | 完全无状态——每次启动从 API 拉取，无本地缓存或数据库 |
| 配置管理 | 所有参数硬编码在模块头部（API URL、并发数、质量阈值等），无配置文件 |
| 输出格式 | HTML（交互式报表）、Excel、TMX（浏览器端导出） |
| 技术栈 | Python 3.12 + tkinter + PyInstaller，依赖仅 `requests` / `openpyxl` |
| 与主平台交互 | 只读调用 `/api/v1/legacy` 和 `/api/v1` 的 GET 端点 |

---

## Phase 1：本地智能层 — "记住我的工作上下文"

**核心价值跃迁**：从"每次启动都是白纸"到"带记忆的个人工作环境"，通过本地持久化让工具成为 PM 的专属工作空间。

### 功能清单

| 功能 | 解决的痛点 | 交互形式 | 对主库依赖 | 复杂度 |
|------|-----------|---------|-----------|--------|
| **配置文件系统** | 每次启动都要重新选择项目/语言/阈值；更换部署环境需改代码 | GUI 设置面板 + JSON 配置文件 | 独立 | Low |
| **本地 SQLite 缓存** | 重复拉取相同数据浪费时间；无法离线查看历史数据；无法做跨时间段对比 | 自动（后台增量同步） | 需要 API（现有端点即可） | Medium |
| **项目看板视图** | MR Pipeline 标签页只能按单维度筛选，缺乏"我负责的项目"聚合视图 | GUI 新标签页（看板卡片式） | 需要 API（`/dashboard/overview` + `/dashboard/filters`） | Medium |
| **快捷收藏与标记** | 发现质量问题后无法标记跟进，下次启动就忘了 | GUI 右键菜单 + 本地标记存储 | 独立 | Low |
| **启动时数据预取** | 每次切换标签页都要等待加载，操作割裂 | 自动（后台线程池在启动时预取常用数据） | 需要 API（现有端点） | Low |

### 技术实现要点

```
关键架构决策：
├── 配置系统
│   ├── ~/.tranzor-exporter/config.json（全局偏好）
│   ├── 支持多环境 profile（dev / staging / prod）
│   └── GUI 设置面板：tkinter Toplevel 弹窗，保存即生效
│
├── 本地缓存层
│   ├── SQLite 数据库：~/.tranzor-exporter/cache.db
│   ├── 表结构映射 API 响应：tasks / translations / evaluations / mr_results
│   ├── 增量同步策略：按 created_at 水位线拉取新增数据
│   ├── 缓存失效：手动刷新 或 TTL（默认 4h）
│   └── Python 标准库 sqlite3，零额外依赖
│
└── 看板视图
    ├── 从 /dashboard/overview 拉取任务状态分布
    ├── 按用户配置的"关注项目列表"过滤
    └── tkinter Canvas 绘制卡片式布局
```

### 阶段里程碑

- [ ] 启动时自动加载上次的项目/语言/日期偏好
- [ ] 离线模式下可浏览上次同步的历史数据
- [ ] "我的项目"看板展示所有关注项目的实时状态
- [ ] 收藏/标记的翻译条目跨会话保持

## Phase 2：分析工作台 — "用数据讲故事"

**核心价值跃迁**：从"查数据"到"看趋势、找规律、生成汇报"，让 PM 能用数据驱动质量改进和向上汇报。

### 功能清单

| 功能 | 解决的痛点 | 交互形式 | 对主库依赖 | 复杂度 |
|------|-----------|---------|-----------|--------|
| **质量趋势仪表盘** | Quality Overview 只展示当前快照，无法看到质量随时间的变化趋势 | GUI 新标签页（折线图 + 对比选择器） | 独立（基于本地缓存） | Medium |
| **跨项目基准对比** | 无法回答"哪个项目质量最差""哪个语言最需要关注" | GUI 仪表盘（雷达图 / 柱状图） | 独立（基于本地缓存） | Medium |
| **效率指标追踪** | 无法量化翻译效率——平均耗时、精炼率、人工修改率等 KPI | GUI 指标卡片 + 趋势线 | 需要 API（`/dashboard/overview` 的 `avg_execution_time`） | Medium |
| **一键生成周报/月报** | 每周手动整理数据做汇报耗时，且容易遗漏 | GUI 按钮 → 自动生成 HTML/Excel 报告 | 独立（基于本地缓存） | Medium |
| **异常检测告警** | 质量突然下降或任务批量失败时，PM 往往事后才发现 | GUI 通知栏 + 可选桌面通知 | 独立（基于本地缓存 + 统计算法） | High |

### 技术实现要点

```
关键架构决策：
├── 可视化引擎
│   ├── 方案 A（推荐）：tkinter Canvas 自绘图表
│   │   ├── 折线图、柱状图、饼图：已有 quality_overview.py Canvas 绘图基础
│   │   ├── 优势：零依赖，打包体积不变
│   │   └── 劣势：交互性有限（无缩放/拖拽）
│   ├── 方案 B（备选）：matplotlib 嵌入 tkinter
│   │   ├── FigureCanvasTkAgg 嵌入
│   │   ├── 优势：图表种类丰富，支持导出 PNG/SVG
│   │   └── 劣势：增加 ~30MB 打包体积
│   └── 建议：Phase 2 先用 Canvas 自绘，需求超出时再引入 matplotlib
│
├── 分析引擎
│   ├── 时序聚合：按 day / week / month 从 SQLite 聚合
│   ├── 滑动窗口：7 日均线、30 日均线平滑噪声
│   ├── 异常检测：简单 Z-score（均值 ± 2σ 标记异常点）
│   └── 基准对比：项目间 percentile 排名
│
├── 报告生成器
│   ├── Jinja2 模板引擎（~100KB 依赖，成熟稳定）
│   ├── 预设模板：周报、月报、项目质量卡
│   ├── 输出：HTML（可直接发邮件）+ Excel（附带图表）
│   └── 数据源：本地 SQLite 聚合查询
│
└── 通知系统
    ├── 应用内：tkinter 顶部通知栏（自动消失 / 点击展开详情）
    ├── 桌面通知：plyer 库跨平台通知（可选依赖）
    └── 触发条件：质量均分跌破阈值、失败任务数突增、新任务完成
```

### 阶段里程碑

- [ ] 打开"趋势"标签页可看到过去 30 天各项目的质量评分走势
- [ ] 选择两个项目可并排对比质量和效率指标
- [ ] 一键生成本周翻译报告（HTML 格式），包含关键指标、趋势、异常高亮
- [ ] 质量评分跌破配置阈值时，顶部通知栏弹出告警

## Phase 3：主动质量管理 — "在工具里闭环问题"

**核心价值跃迁**：从"被动查看数据"到"在工具中直接处理质量问题"，将发现→诊断→修复→验证的闭环内化到 Exporter 中。

### 功能清单

| 功能 | 解决的痛点 | 交互形式 | 对主库依赖 | 复杂度 |
|------|-----------|---------|-----------|--------|
| **翻译审校工作流** | 发现低分翻译后需要切到 Tranzor 网页端逐条处理，上下文割裂 | GUI 审校面板（源文 / 译文 / 评分 / 评语并排展示，直接编辑） | 需要 API（`PUT /dashboard/comment`、`POST /dashboard/fix-translation`） | High |
| **批量重译与引导** | 多条低质量翻译需要逐一提交重译请求 | GUI 多选 + 批量操作栏 | 需要 API（`POST /dashboard/translations/retranslate-batch`） | Medium |
| **术语一致性检查器** | 无法系统性地检测同一术语在不同翻译中的一致性 | GUI 检查报告 + 高亮不一致项 | 独立（基于本地缓存中的翻译数据做文本匹配） | Medium |
| **翻译搜索与交叉引用** | 想查某个 key 或某段文本在不同项目/MR 中的翻译情况，无处可查 | GUI 搜索面板（支持 opus_id、源文、译文模糊搜索） | 需要 API（`GET /translations/search`） | Low |
| **问题追踪看板** | 标记的问题翻译缺乏状态管理（待处理/已修复/已验证） | GUI 看板（拖拽卡片式或列表+状态列） | 独立（本地 SQLite 存储问题状态） | Medium |

### 技术实现要点

```
关键架构决策：
├── 审校面板
│   ├── 布局：左侧源文+译文并排，右侧评分详情+错误类别
│   ├── 编辑模式：双击译文区域进入编辑，保存时调用 fix-translation API
│   ├── 批量模式：Checkbox 多选 → 底部操作栏（重译/导出/标记）
│   ├── 键盘快捷键：↑↓ 导航、Enter 编辑、Ctrl+S 保存、Esc 取消
│   └── 注意：这是首次引入写操作（POST/PUT），需要认证支持
│
├── 认证模块（新增，Phase 3 前置条件）
│   ├── POST /auth/login 获取 JWT token
│   ├── token 存储：keyring 库（OS 原生密钥链）或加密本地文件
│   ├── 自动续期 / 过期重登
│   └── 配置面板新增用户名/密码输入
│
├── 术语一致性检查
│   ├── 数据源：本地缓存中所有译文，按 opus_id 前缀归组
│   ├── 算法：同一英文术语 → 收集所有目标语言译文 → 标记偏差
│   │   ├── 精确匹配：相同源文片段的不同译法
│   │   └── 模糊匹配：编辑距离 ≤ 2 的近似术语
│   ├── 输出：不一致报告（术语 / 出现次数 / 各译法及使用频率）
│   └── 未来可对接主平台术语库 API（如有）
│
└── 问题追踪
    ├── SQLite 新表：issues (id, translation_id, status, notes, created_at, resolved_at)
    ├── 状态流转：Open → In Progress → Fixed → Verified
    ├── 与收藏系统合并：Phase 1 的收藏升级为问题条目
    └── 支持按状态/项目/语言筛选
```

### 阶段里程碑

- [ ] 登录主平台后可在 Exporter 中直接修改翻译并提交到 GitLab
- [ ] 筛选低分翻译 → 多选 → 一键批量重译，带自定义引导指令
- [ ] 术语一致性检查报告输出：覆盖所有缓存翻译，按不一致严重程度排序
- [ ] 问题看板展示所有标记的翻译问题，可追踪从"发现"到"验证"的完整生命周期

## Phase 4：AI 增强与自动化 — "让工具替我巡检"

**核心价值跃迁**：从"人驱动工具"到"工具辅助人"，利用 LLM 和自动化让日常巡检、分析、汇报等重复性工作自动发生。

### 功能清单

| 功能 | 解决的痛点 | 交互形式 | 对主库依赖 | 复杂度 |
|------|-----------|---------|-----------|--------|
| **自然语言质量摘要** | 数字指标需要人工解读，写汇报时要手动归纳趋势 | GUI "生成摘要" 按钮 → AI 生成的文字段落 | 独立（调用 Claude API 或 OpenAI） | Medium |
| **定时巡检与报告推送** | PM 需要每天手动打开工具检查，容易遗漏 | 后台定时任务 + 邮件/企业微信推送 | 需要 API（现有端点） | High |
| **智能问题诊断** | 低分翻译需要人工判断原因（术语错误？漏译？风格偏差？），效率低 | GUI 诊断面板（AI 自动分析错误根因并给出修复建议） | 独立（调用 LLM API） | Medium |
| **翻译质量预测** | 只能事后看到质量分数，无法提前预判哪些 MR 可能出问题 | GUI 风险指示器（MR 列表中标注预测风险等级） | 独立（基于本地历史数据训练简单模型） | High |
| **自定义规则引擎** | 不同项目有不同质量标准和检查规则，目前只能用统一阈值 | GUI 规则编辑器（条件 + 动作，类似邮件规则） | 独立 | Medium |

### 技术实现要点

```
关键架构决策：
├── LLM 集成
│   ├── 统一接口层：支持 Claude API / OpenAI / 本地模型
│   ├── 提示词模板化：Jinja2 渲染提示词，可用户自定义
│   ├── 用途：
│   │   ├── 质量摘要：传入聚合指标 → 生成自然语言段落
│   │   ├── 问题诊断：传入源文/译文/评分/错误类别 → 分析根因
│   │   └── 报告生成：传入周期数据 → 生成带洞察的完整报告
│   ├── 成本控制：缓存相同输入的 LLM 响应
│   └── API Key 存储：复用 Phase 3 的 keyring 密钥管理
│
├── 定时任务系统
│   ├── 方案 A（推荐）：APScheduler 库（轻量，嵌入式）
│   │   ├── 支持 cron 表达式和间隔触发
│   │   ├── 持久化调度到 SQLite
│   │   └── 优势：不依赖 OS 任务计划，打包即可用
│   ├── 触发链：定时同步数据 → 运行检查规则 → 生成报告 → 推送通知
│   └── 推送渠道：
│       ├── 邮件：smtplib（标准库）
│       ├── 企业微信/Slack：Webhook HTTP POST
│       └── 桌面通知：复用 Phase 2 的 plyer
│
├── 质量预测模型
│   ├── 特征工程：项目ID、语言、源文长度、变量数、HTML标签数、历史均分
│   ├── 模型：scikit-learn 决策树 / 随机森林（轻量，可解释）
│   ├── 训练数据：本地 SQLite 中的历史翻译+评分
│   ├── 推理：新 MR 进入时实时计算风险分数
│   └── 注意：打包体积增加 ~50MB（numpy + scikit-learn）
│       → 可选：用纯 Python 实现简单决策树避免大依赖
│
└── 规则引擎
    ├── 数据模型：rules (id, name, conditions_json, actions_json, enabled)
    ├── 条件类型：score </>= N, language in [...], project in [...], error_category = X
    ├── 动作类型：标记问题、发送通知、加入报告、自动重译
    ├── GUI 编辑器：类似邮件规则——添加条件行 + 选择动作
    └── 执行时机：数据同步后自动运行所有启用的规则
```

### 阶段里程碑

- [ ] 点击"生成摘要"可获得本周翻译质量的自然语言分析
- [ ] 配置每日 9:00 自动巡检，异常时通过企业微信推送告警
- [ ] 低分翻译可一键 AI 诊断，获得错误根因和修复建议
- [ ] MR 列表中显示质量风险预测指标，高风险项红色标注

### 飞轮加速的关键拐点

| 拐点 | 触发条件 | 效果 |
|------|---------|------|
| **数据积累拐点** | 本地缓存积累 > 3 个月历史数据 | 趋势分析从"噪声"变为"信号"，异常检测误报率显著下降 |
| **行为闭环拐点** | Phase 3 审校工作流上线 | PM 的修复行为直接产生训练数据，不再需要去主平台操作 |
| **自动化拐点** | Phase 4 定时巡检 + 规则引擎上线 | 日常巡检从"人推动"变为"系统推动人确认"，PM 时间释放到策略层面 |
